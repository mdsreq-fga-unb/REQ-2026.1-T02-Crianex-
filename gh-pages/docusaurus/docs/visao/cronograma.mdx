Na IT1 ocorreu **duas vezes**: a primeira está registrada na [ata de 14/05](/iteracoes/iteracao-1/atas/ata_refinamento_requisitos) (com gravação); a **segunda ocorreu, porém sem gravação** — esquecemos de registrá-la em vídeo. Ambas refinaram requisitos e revisaram a matriz de rastreabilidade. Os **artefatos** entregues na iteração (diagramas de sequência, arquitetura, Resultados V&V) ficam na coluna _Resultados do ciclo_, separados das features.
:::

<ProcessFlows />

### Regras do fluxo

| Regra | Descrição |
| ----- | --------- |
| **Sem atalhos** | Uma issue não pode pular colunas no Kanban (ex.: In Progress direto para Done). |
| **WIP limit é lei** | Máx. 2 issues In Progress por Class Owner; ao atingir o limite, ajude a destravar antes de iniciar nova. |
| **Design antes de código** | Nenhuma linha é escrita sem Technical Design Review e critérios de aceite definidos. |
| **Entrega contínua** | Features são entregues a Otávio conforme ficam prontas — sem esperar a data da unidade acadêmica. |
| **Rastreabilidade mandatória** | Toda issue é linkada à Feature parent no Miro e à CP no Documento de Visão. |

## Marcos e Critérios de Encerramento

Cada iteração só é **encerrada** quando todos os critérios abaixo estão satisfeitos:

| # | Critério | Responsável |
| - | -------- | ----------- |
| 1 | Issues comprometidas em Done ou com carry-over documentado | Development Manager |
| 2 | Formal Client Validation realizada e aprovação registrada na ata | Responsável por Validação |
| 3 | Matriz de rastreabilidade OE → CP → Feature → Issue → PR → Validação atualizada | Documentation Lead |
| 4 | Documento de Visão (Pages) atualizado com artefatos da iteração | Documentation Lead + PM |
| 5 | Retrospectiva realizada e lições aprendidas registradas | Facilitador Metodológico |
| 6 | Backlog macro reordenado por IP para a próxima iteração | Project Manager |
